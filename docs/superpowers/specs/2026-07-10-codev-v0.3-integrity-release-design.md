# CO-DEV v0.3 Integrity Release Design

Date: 2026-07-10
Status: Approved for implementation

## Purpose

CO-DEV v0.3 strengthens the integrity of the existing lightweight governance model without adding more skills or replacing the single-file `.codev.md` format.

The release addresses five concrete gaps:

1. An approval can currently be reused after `Current gate` changes.
2. Required state fields and state combinations are not fully validated.
3. The CO-DEV repository is not governed by its own `.codev.md`.
4. Plugin manifests still report `0.1.0` while the current structure is described as v0.2.
5. The checker and CI are Windows-first rather than cross-platform.

## Product Constraints

- Keep `.codev.md` as the default project state file.
- Keep `Intent + Shape + Trace` as the default document structure.
- Keep the four-skill model: `using-codev`, `codev-shape`, `codev-gate`, and `codev-drift`.
- Keep human approval as the governing decision. Automated checks remain evidence.
- Do not introduce a hosted service, database, cryptographic identity system, or document farm.
- Preserve the existing PowerShell checker command as a compatibility entry point.

## Approaches Considered

### A. Patch the PowerShell checker only

Add `Decision gate` and more regular-expression checks to `codev-check-gate.ps1`.

This is the smallest change, but it leaves the state model tied to Windows-oriented tooling and duplicates future parsing logic when another CLI is added.

### B. Canonical Python CLI with a PowerShell compatibility wrapper

Create one dependency-free Python CLI that owns parsing, validation, status, checking, and approval updates. Keep `codev-check-gate.ps1` as a thin adapter.

This provides one state implementation, works on Windows, macOS, and Linux, and retains the existing command for current users.

### C. Replace Markdown state with JSON or YAML

Move runtime metadata into a structured state file and keep narrative sections separately.

This would simplify parsing but breaks the single-file, human-readable product shape and creates unnecessary migration cost.

## Selected Approach

Approach B is selected.

Python 3.9 or newer is the cross-platform runtime. The implementation uses only the standard library. PowerShell remains a compatibility launcher and does not implement state semantics independently.

## State Format

The metadata preamble appears before the first level-two Markdown heading:

```markdown
# CO-DEV

Gate: normal
Ceremony: light
Execution engine: superpower
Current gate: integrity-release-v0.3
Decision: pending
Decision gate: integrity-release-v0.3

## Intent
...
```

The required fields are:

| Field | Valid values |
|---|---|
| `Gate` | `ultra`, `strict`, `normal`, `loose`, `free` |
| `Ceremony` | `light`, `standard`, `audit` |
| `Execution engine` | `default`, `superpower`, `codex`, `cursor`, or `custom:<non-empty-name>` |
| `Current gate` | `none` or a non-empty gate identifier |
| `Decision` | `pending`, `approved`, `redirected`, `rejected`, or an existing compact approval synonym |
| `Decision gate` | `none` or a gate identifier |

Only the metadata preamble is parsed. Field-like text inside `Intent`, `Shape`, `Trace`, examples, or fenced code blocks is not state.

Each required field must appear exactly once. Missing and duplicate fields are invalid.

## State Invariants

### No active gate

When `Current gate: none`:

- `Decision` must be `pending`.
- `Decision gate` must be `none`.
- `check` succeeds because there is no active approval boundary.

### Active gate

When `Current gate` names a gate:

- `Decision gate` must exactly equal `Current gate`.
- `pending`, `redirected`, and `rejected` block continuation.
- `approved` and existing approval synonyms pass only when the gate identifiers match.
- `Gate: free` still validates the complete state and gate binding before returning the low-assurance pass result.

Changing `Current gate` without changing `Decision gate` therefore produces an invalid state instead of reusing an old approval.

## CLI

The canonical entry point is:

```text
python scripts/codev.py <command> [options]
```

### `check`

```text
python scripts/codev.py check --project-root .
```

Validates the state and returns:

- `0` when continuation is permitted.
- `1` when state is valid but the human gate blocks continuation.
- `2` when state is missing or invalid.

### `status`

```text
python scripts/codev.py status --project-root .
```

Validation runs before status output. Invalid state returns `2`; status never reports malformed state as healthy.

### `approve`

```text
python scripts/codev.py approve --project-root . --gate integrity-release-v0.3
```

Approval requires the human or caller to name the gate explicitly. The command:

1. Validates the existing state.
2. Requires `--gate` to exactly match `Current gate`.
3. Writes `Decision: approved`.
4. Writes `Decision gate: <Current gate>`.
5. Preserves the rest of `.codev.md`.

A mismatched gate fails without modifying the file.

## PowerShell Compatibility

`scripts/codev-check-gate.ps1` remains supported:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\codev-check-gate.ps1 -ProjectRoot .
powershell -ExecutionPolicy Bypass -File .\scripts\codev-check-gate.ps1 -ProjectRoot . -Status
```

The wrapper locates a Python 3 interpreter and delegates to `scripts/codev.py`. It forwards the Python exit code unchanged. If Python is unavailable, it returns `2` with an actionable message.

## Repository Self-Governance

The `codev` repository receives its own `.codev.md`:

- `Gate: normal`
- `Ceremony: light`
- `Execution engine: superpower`
- `Current gate: integrity-release-v0.3`
- `Decision: pending`
- `Decision gate: integrity-release-v0.3`

CO-DEV-specific trace entries currently stored in `D:\Line\.codev.md` are migrated into `D:\Line\codev\.codev.md` and removed from the Line Plan state file.

## Versioning

This is a schema and tooling release, so all plugin manifests and marketplace metadata move to `0.3.0`:

- `.codex-plugin/plugin.json`
- `.claude-plugin/plugin.json`
- `.cursor-plugin/plugin.json`
- `.claude-plugin/marketplace.json`

README examples and the default template include `Decision gate`.

## Tests

The cross-platform Python tests cover:

- A new gate cannot reuse an old approval.
- Matching approval and decision gate passes.
- Missing required fields fail.
- Duplicate required fields fail.
- Invalid gate, ceremony, execution engine, decision, and state combinations fail.
- `status` fails on malformed state.
- `free` mode still validates state integrity.
- `approve` updates both decision fields.
- `approve` with the wrong gate leaves the file unchanged.
- Field-like text after the metadata preamble is ignored.
- Manifest versions are consistently `0.3.0`.
- The repository template contains every required field.

Windows compatibility tests continue to exercise the PowerShell wrapper.

## CI

GitHub Actions gains:

- A Python test matrix for `windows-latest`, `ubuntu-latest`, and `macos-latest`.
- A Windows compatibility job for the existing PowerShell structure and wrapper tests.
- Plugin manifest validation using the available plugin validator.

## Migration Behavior

Existing state files without `Decision gate` fail with an actionable migration message:

```text
Missing 'Decision gate' in .codev.md. Set it to 'none' or the active Current gate.
```

The checker does not silently infer an approved gate because inference would preserve the stale-approval vulnerability.

## Non-Goals

- Proving cryptographically that a human, rather than an agent, ran `approve`.
- Adding remote approvals, accounts, signatures, or an audit server.
- Adding more workflow skills.
- Replacing Markdown with JSON or YAML.
- Automatically selecting gate boundaries from source-code changes.

## Definition of Done

- The stale-approval reproduction returns a non-zero exit code.
- All six metadata fields are validated before every command.
- The canonical parser is shared by `check`, `status`, and `approve`.
- The PowerShell entry point delegates to the canonical CLI.
- The repository has its own valid `.codev.md`.
- Line Plan no longer contains the migrated CO-DEV trace entries.
- Plugin versions are consistently `0.3.0`.
- Python tests pass on Windows, macOS, and Linux CI runners.
- PowerShell compatibility tests pass on Windows.
- The Codex plugin manifest passes the official local validator.
